This page will instruct you on how to make object types in Pikifen. These include types of enemies, types of Pikmin, etc.

General idea

In order for the engine to detect a type of object, it will check the game_data/<pack>/mob_types folders. Inside are several subfolders, one for each category of object (enemies, gates, etc.). Inside each category are folders for the different types of objects (the different types of enemies, different types of leaders, etc.). Each type here is a piece of content, so you should know how content works in the engine in general before starting. Finally, inside those are some data file with data about how the object works, its animations, or its script.

Creating a new object type

To create a new object type, make a folder for it in the right category, with the folder's name representing the object type's internal name. Create an empty data file inside called data.txt. Inside of that, you will write all of the object's properties, like the name, its movement speed, etc.

Some properties are mandatory, some are optional, some apply to all object types, and some only apply to certain categories of object type. You can get a list of properties that apply to all object types in the following sub-sections of this page, and then check what properties exist for that category by visiting the category's article from the list. After you're done setting the properties, you must create the animations for the object type; use the animation tutorial for this. Finally, some object types will also need a script to run. To figure out if your object will need a script or not, check the list of categories and if it does, read the object script tutorial to learn what you need to do.

After you have done that, you can boot up the engine, go to the area editor, place your new object in an area, hit play, and watch it in action!

Naturally, instead of creating your own files from scratch, you may copy and paste an existing object's files and simply edit those.

Common content properties

See here for a list of basic properties a mob type can use. For the description, this should be written from a gameplay perspective, not a content-making perspective, so describe the object how it works in-game, not how it should be used in areas or with other objects. This description is useful for whoever reads the data file, but also shows up in the area editor, and in the case of Pikmin types, the help menu. You can use \n to cause a line break and start a new line. To write a literal backslash, you will need to type \\.

Recommended properties

The following are properties that you really should have, but the engine can run without them.

can_hunt This object will only want to hunt down objects of this target type. i.e. it will only consider those an "opponent" (provided they're also in an opposing team). See target_type below for more information. Text list player; enemy
can_hurt This object can only cause damage to objects of this target type (provided they're also in an opposing team). See target_type below for more information. Text list player; enemy; fragile
height Object's height. This doesn't need to be very specific. For reference, leaders are usually 32, Pikmin are usually 24. Number 0
max_health The object's top health. Use this responsibly, since players are used to seeing specific enemies with specific health amounts and plan accordingly, and if you change this their plans will be foiled. For reference, leaders have 2000, and something like a Red Bulborb has 750. Number 100
move_speed Standard movement speed (in pixels per second). For reference, leaders move at 130. Number 0
pushable true if this object can be pushed out of the way by other objects. false otherwise. Boolean false
pushes true if this object can push "pushable" objects out of the way. false otherwise. See also: pushes_softly. Boolean false
radius The object's size, represented by the circle's radius (in pixels). Leaders are 32 pixels wide, so their radius is 16. Pikmin have a radius of 10. Number 0
rotation_speed How fast the object turns when moving, in degrees per second. For reference, leaders use around 360, Pikmin use around 630, and a Red Bulborb uses around 114. Number 630
target_type What type of "target" the object is. This determines which objects will hunt it, and which objects can attack it. Possible values are as follows, along with what each type is meant to represent.
  • none: Cannot be damaged or hunted down.
  • player: Leaders, Pikmin, and anything belonging to the "heroes" of the player's side.
  • enemy: Enemies.
  • weak_plain_obstacle: Weaker objects that should be damaged by many things, like leaders, Pikmin, or explosions. e.g. egg.
  • strong_plain_obstacle: Stronger objects that should be damaged by beefy hits, like Pikmin or explosions. e.g. gas pipe.
  • pikmin_obstacle: An object only Pikmin should harm. e.g. electric gate.
  • explodable: An object that should only be hurt by an explosion. e.g. reinforced wall.
  • explodable_pikmin_obstacle: An object that should be hurt by Pikmin or explosives only. e.g. Burgeoning Spiderwort.
  • fragile: A fragile object that should be harmed by even the gentlest smack. e.g. bomb rock.
 Text none
team Sets the object's team. Objects in the same team will not hunt or harm each other. Default depends on the object's category. Valid values are:
  • none: Does not belong to a team.
  • player_X: Belongs to player 1, 2, 3, or 4's team (replace the X with a number). Usually used for leaders and Pikmin.
  • enemy_X: Belongs to the enemy team 1, 2, 3, or 4 (replace the X with a number).
  • obstacle: Obstacle team.
  • other: Team for whatever else.
 Number

Optional properties

For all categories of object type, you may set these properties, but they will still work if you don't.

acceleration How quickly the object reaches top speed when moving. This is in units per second per second. Number 400
blackout_radius How much light it emits during blackout weather conditions. 0 means no light, and is the default for decorations. -1 means it uses the object's own radius, which is the default. Number
can_block_paths If set to true, this object blocks objects following the path it is on, like a closed gate or an unbuilt bridge. If the object finishes dying, it will stop blocking, though it can also start and stop blocking at will according to its script. Boolean false
can_walk_on_others If true, this object can move in any direction freely, without having to first turn there. It will also not face towards the direction it is going. Boolean false
can_free_move If true, this object can walk on top of objects that have the "walkable" property. Boolean false
casts_shadow If set to false, the object won't cast a shadow on the ground. Boolean true
custom_carry_spots Normally Pikmin place themselves around the object in a ring pattern when carrying it. With this you can specify custom positions. This is a list of relative coordinates, e.g.: 128 0; 100 50; -30 -30. Coordinate list
custom_category_name The categories that object types belong to have names that describe how they work fairly well, but aren't very obvious otherwise. To make life easier for people trying to search for this object type inside an editor, you can use a custom category name. Text
default_vulnerability For every source of damage not specified in the vulnerabilities, the damage the object receives is multiplied by this amount. Number 1
has_group If true, this object can have a group of other objects (usually a group of Pikmin) following behind it, likely via the follow_link_as_leader script variable. Without this property, it can't have other objects following it. Boolean
health_regen Passive health regeneration rate, in HP per second. Number 0
inactive_logic Controls how the object works when it is not active. normal by default, except for Pikmin and leaders, where it's all_logic by default. This property is fairly important for performance. Possible values are:
  • normal: Normal logic – the object does nothing when inactive.
  • all_logic: The object still does its whole logic (i.e. ticks and interactions) when inactive.
  • ticks: The object still ticks its frame-to-frame logic when inactive.
  • interactions: The object still interacts with other objects when inactive.
 Text
itch_damage If an enemy has taken this much damage, it will be considered "itchy". With this, you can listen to the on_itch event, and when it triggers, make the enemy shake the Pikmin off. Handling this event automatically resets the amount of damage, making it possible for the enemy to be itchy again, next time it takes this amount of damage. Setting this to something like 10% or 20% of the max health is usually the way to go. Number 0
itch_time In order for the enemy to not shake over and over, if you keep damaging it, you can also specify a minimum time requirement between itches, in seconds. Number 0
main_color A color to represent the object by. Color 0 0 0
max_carriers Maximum number of Pikmin that can carry this object, when it is made carriable. Number 0
pushes_softly If true, and if this object can push others, then the push is fairly soft, allowing leaders and Pikmin to squeeze through if they persist for long enough. Boolean false
pushes_with_hitboxes If true, this object pushes with its (normal or attack) hitboxes, instead of pushing using its object's center and radius. Boolean false
rectangular_dimensions If this object is meant to have rectangular collision, specify the rectangle's dimensions here, in the format <width> <height>. This is useful for objects like gates. Coordinates 0 0
show_health If set to false, the object's health wheel will not appear on top of it. Boolean true
spike_damage If this object is meant to cause spike damage, specify the spike damage type here. Internal name
spike_damage_vulnerabilities Normally, objects hit by a spike damage attack will receive the damage specified by the attack's type. If you want this mob type to take more or less damage instead, or even receive a status effect, you can specify it here. This property is a block, and the format is the same as the regular damage vulnerabilities property. Block
status_vulnerabilities If you want this mob type to take more or less damage from a status effect's health changing effect, or even receive a different status effect instead, you can specify it here. This property is a block, and the format is the same as the regular damage vulnerabilities property, without the third word. Block
terrain_radius The object's radius for collisions against the terrain. Useful if you want collisions between your object and other objects to be tight, but you also want your object to not get too close to walls so it doesn't visually appear inside them. Any negative number will make the engine use the same radius as the radius property. This property is only used while the object is alive, so a carriable corpse will use the regular radius property. Number -1
territory_radius How far away from the spawn point must the enemy be before being considered "far from home". For reference, a Red Bulborb has this set to 90. Number 0
walkable If true, this object can be walked on top of by other objects. Other objects will be able to mount it if they fall on it from above, or if they are on a floor that's within 50 units of height from the top of this object (much like how climbing up stairs works). Boolean false
weight How much Pikmin strength does it take to carry the object. Number 0

Extra features

Reaches

An object can detect whether another object is "within reach" or not. Also, if it was within reach, and became "out of reach". It can also use different definitions of "reach" for different moments. This is all explained in detail in the script page.

Spawning other objects

If you want your object to spawn other objects, you need to specify their spawn data. In data.txt, include a block titled spawns. Inside, you will write a series of blocks, each one with a unique name that will identify that spawn data. Later, when you want to spawn an object using the script, you need to refer to this name.

These spawn data blocks must have the following properties inside:

object Object type to be spawned. Internal name

It can also optionally have these properties:

relative Are the spawn coordinates specified in the coordinates property absolute or relative? Boolean true
coordinates Coordinates to spawn in. Whether they are relative or absolute depends on the relative property. Text 0 0 0
angle Absolute angle the spawned object will be facing. Number 0
momentum If this is 0, the mob spawns in the exact spot specified and stand there. Otherwise, it will spawn in that spot but be launched in a random direction, with the specified momentum. 100 is a good general case value. Number 0
vars Any script variables to include, separated by semicolon. Example: vars = maturity = 0; sprout = true. List
link_spawn_to_object If true, a link will be created from the spawned object to your object. Boolean false
link_object_to_spawn If true, a link will be created from your object to the spawned object. Boolean false

Parent/child objects

Your object can be a parent object, meaning it is actually composed of several different child objects. This feature is especially useful when you need several independently-moving parts. For instance, in a Beady Long Legs, the head is the parent object, meaning it is the enemy proper, and each leg is a child object.

A parent will spawn its children at the same time that it itself is spawned. Because every child is meant to be its own individual thing but also serve as components of the parent object, there are several ways for you to specify what it should do when it interacts with the world. This means specifying what happens when it receives damage, receives a script event, or receives a status effect: it can either ignore what it received or not, and it can either relay it to the parent object or not.

Children objects can also be connected to a specific body part in the parent. Because children objects are commonly used as appendages of the creature, it is possible to draw a "limb" connecting a parent object to a child object. This limb is a rectangular shaped sprite that stretches and rotates accordingly.

To make your object type a parent, go to data.txt, and add a block called children. Inside, you will include a series of blocks, each one with a unique name that represents the child, e.g. "left_foot". Now, when your object is spawned, it will spawn these children along with it. Each of these children blocks needs to have the following properties:

spawn Name of the spawn data to use. Text

The blocks can also have the following optional properties:

parent_holds If true, then the parent object is holding this one, much like an enemy holds a Pikmin that is latched on to it. Use the following properties to specify the location. Boolean false
hold_body_part If the parent holds this child, this is the name of the body part it is held on. If empty, the center of the parent object is used instead. Text
hold_offset_distance If the parent holds this child, this is how far away from the center of the body part/object the child is. 0 means dead center, and 1 means on the edge, but you can write numbers beyond this range. Number 0
hold_offset_vertical_distance If the parent holds this child, this is how far away from the bottom of the body part/object the child is. 0 means same as bottom, and 1 means same as top, but you can write numbers beyond this range. Number 0
hold_offset_angle If the parent holds this child, this is the relative angle from the body part/object's default position that the child is located at. Number 0
hold_rotation_method Indicates how the child can rotate. never means it never rotates (besides the starting angle). face_parent means it always rotates to face the center of the parent object. copy_parent means it's always the same angle as the parent object. Text never
handle_damage If true, the child object loses health when it receives damage. If false, it ignores the damage. Boolean false
relay_damage If true, the child object relays any damage to the parent object, so it can handle it. Boolean false
handle_events If true, the child object handles any script event that it receives. If false, it ignores the event. Boolean false
relay_events If true, the child object relays any events received to the parent object, so it can handle them. Boolean false
handle_statuses If true, the child object handles any status effect that it receives. If false, it ignores the status. Boolean false
relay_statuses If true, the child object relays any status effects received to the parent object, so it can handle them. Boolean false
limb_animation If there is meant to be a "limb" drawn between the child and parent, specify the name of its animation here. This animation must be included in the parent object's animation set. Text
limb_thickness If there is a limb, this specifies how thick it is. Number 32
limb_parent_body_part If there is a limb, this specifies where it connects to in the parent. If it's empty, it connects to the parent's center. Otherwise, it connects to the center of the specified body part. Text
limb_parent_offset If there is a limb, you can specify how far away it is from the center of the parent-side connection. Number 0
limb_child_body_part Same as limb_parent_body_part, but for the child object. Text
limb_child_offset Same as limb_parent_offset, but for the child object. Number 0
limb_draw_method How to draw the limb. Defaults to above child. This can be one of the following:
  • below_both: Draw the limb below both the parent and the child.
  • below_child: Draw the limb below the child. It may appear above or below the parent, depending on the child and parent's positions.
  • below_parent: Draw the limb below the parent. It may appear above or below the child, depending on the child and parent's positions.
  • above_parent: Draw the limb above the parent. It may appear above or below the child, depending on the child and parent's positions.
  • above_child: Draw the limb above the child. It may appear above or below the parent, depending on the child and parent's positions.
  • above_both: Draw the limb above both the parent and the child.
 Text above_both

Vulnerabilities

You can make it so that an object is completely invulnerable to a hazard, more vulnerable to another, etc.

In data.txt, add a block titled vulnerabilities. Inside, each property has the hazard's internal name as its name, and the vulnerability data as its value, e.g. fire = 50 sweating true. The vulnerability data consists of one to three words:

  1. The first word is the effect percentage. When the object is hurt by an attack with this hazard, its damage will be changed by the specified percentage, so 50 would make it take half damage. You can make this negative to get the object to heal instead! This value also affects some other things about the hazard's status effects, like speed reductions. If this value is 0, the object will completely ignore this hazard in its entirety, even if it's not an attack.
  2. The second word, optional, is the internal name of a status effect. When the object interacts with this hazard, it will receive this status effect.
  3. The third word, also optional, controls whether the status effect overrides others. If true, then the object will only receive the status effect from the previous word, and will not receive the status effect the hazard usually gives. If false or not specified, then the object will also receive this status effect on top of any others the hazard gives.

You can use this in conjunction with the default_vulnerability property to make an object that is invulnerable to anything except explosions, for instance.

Sounds

To make your object play sounds, you need to specify information about them. Typically you just specify what the sound file is, but more properties exist to make the object play sounds in more interesting ways.

In data.txt, add a block titled sounds. Inside will be a list of blocks, where the name of a block is the name of the sound (which you will use later to refer to this sound). Each sound block must have the following properties.

file Sound to play. This must be in the game_data/<pack>/audio/sounds folders. Internal name

Each block can also have the following properties.

type Type of sound. gameplay_global means it's an in-world sound important to gameplay, but doesn't play from any specific spot in the area. gameplay_pos means it's an in-world sound important to gameplay that does play from a specific spot, meaning it comes from the object itself. ambiance_global means it's in-world, but just a general background ambiance sound. ambiance_pos means it's in-world, coming from a specific point, but it's just ambiance. ui means it's a UI sound, like a chime. Text gameplay_pos
stack_mode If the object needs to play this sound again while another instance of this sound is already playing, what should happen? normal means the second sound plays too, like normal. override means the first sound stops and the second plays instead, overriding it. never means the second sound will never play so long as the first sound is still playing. Text normal
stack_min_pos If the object needs to play this sound again while another instance of this sound is already playing, the second sound is only allowed to stack if the first one is already these many seconds in. Avoid using 0, since doing so can allow for the same sound to play many times at the exact same time, which just results in a really loud sound. Number 0.1
loop If you want the sound to loop, set this to true. Make sure you only use this for sounds that you intend to stop (i.e. via the script actions play_sound and stop_sound), otherwise the sound will keep playing forever! Boolean false
volume Volume it should play at, from 0 to 100. Number 100
speed Speed to play the sound at. The faster it plays, the higher-pitched it sounds too. 100 plays at normal speed, 200 at double-speed, etc. Number 100
volume_deviation Every time the sound plays, deviate the volume up or down by a random amount, between 0 and this number. Number 0
speed_deviation Same as above, but for the speed. Number 0
random_delay Every time the sound is meant to play, wait a random amount of time between 0 and this before actually starting. Number 0

Area editor tips

Some objects require certain things to be set up in certain ways in the area editor before they work as intended. e.g. bridge fragment piles need to know what bridge they belong to, seesaw blocks need to link to each other before they work together, etc. Users who are not very familiar with these tricks may take a while to realize what's wrong. To help with this, you can also specify a property that, when read by the area editor, will make it so users can place their mouse over a text widget, and they will see some helpful tips about how to make use of the object.

In data.txt, add a property titled area_editor_tips. Then, simply write the tip(s) you want as the value. You can use \n to cause a line break and start a new line. To write a literal backslash, you will need to type \\.

You can also add the area_editor_recommend_links_from property and give it the value true so that the area editor warns whenever the maker places this object and it has no links going from it to another object. Likewise, the area_editor_recommend_links_to property makes the editor warn when there are no objects linking to it.

Area editor properties

Some object types can make use of script variables to make specific objects have different behavior. For instance, with script variables, you can place two Pellet Posies in an area, and one of them can start already fully bloomed. When an object spawns, the engine itself or the object's script may read any declared script variables and use them as they see fit. In the area editor, you can write down the script variables you want a specific object to declare, along with their values, but having to memorize the names of the variables – as well as what each value means – is a bit of a pain. To remedy this, you can add some properties to the object type's data with information about important script variables. Then, when the area editor loads, it will use that information to present the user with some helpful widgets to set those special behaviors with, each widget in charge of one script variable.

In data.txt, add a block titled area_editor_properties. Inside, you will write one block per script variable, with each block's name being the display text of the widget responsible. Then, add the following properties:

var The name of the variable, as used in the script. Text

You can also add these properties as needed:

type What type of variable this is. Can be one of the following:
  • text: The variable is meant to be text. The widget will be a text box. Use this when the other types won't cut it, so that the user is free to write whatever they need in the text box.
  • int: The variable is meant to be an integer number. The widget will be a number box with integers.
  • float: The variable is meant to be a float number. The widget will be a number box with float numbers.
  • bool: The variable is meant to be true or false. The widget will be a checkbox.
  • list: The variable is meant to be text. The widget will be a list box with several choices to choose from.
  • number_list: The variable is meant to be an integer number. The widget will be a list box with several choices, where each choice maps to a number.
 Text text
def_value The variable's default value, as used in the script. Text
min_value For variables meant to be integer or float numbers, this is the minimum value it is meant to have, if any. The widget will not let the user pick a number beyond this one. Number
max_value Same as min_value, but for the maximum. Number
list For list widgets, this is the list of choices. If the variable is meant to be an integer, the first option maps to 0, the second to 1, etc. List
tooltip Tooltip to display when the user places their mouse cursor on top of the widget. Use this to explain what the variable means, to provide extra info, to give special instructions, etc. You can use \n to cause a line break and start a new line. To write a literal backslash, you will need to type \\. Text

Notes and tips

  • If objects of your new type are not being hunted down or hurt, confirm the following:
    • The object's target type is one of the types the attacker wants to hunt or can hurt.
    • The object is not in the same team as the attacker.
    • The object has more than zero health.
    • The object is not unhuntable via the set_huntable script action.
    • The object is not invisible via a status effect.
  • If you're having trouble understanding why your custom object is acting the way it is, try aiming your cursor at it and using the "mob info" maker tool (5 key, by default).
  • Avoid making carriable objects with more than 95 units of radius. Gates and most choke points typically have a gap of 192 units, so larger objects likely won't fit through.
